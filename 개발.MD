# Wishket Client Automation Hub — 개발 계획서

> 프로젝트 한 줄 정의: 위시켓 의뢰서를 입력받아 상담 보조 → 메신저 전달 → 자동화 개발 사양화 → 고객 납품까지 연결하는 반자동 운영 하네스

---

## 1. 시스템 구조 개요

### 운영 원칙
- Claude Code = 설계, 구현, 파일 생성, 프롬프트 생성, 상태 문서 갱신
- Codex = 검증, 누락 점검, 오류 탐지, 보안/안전성 검토, 재검증
- LLM Wiki = 현재 상태, 결정 사항, 검증 로그, 다음 작업 저장
- 자동화보다 승인 기반 운영 우선 (MVP 기준)
- 고객 메시지는 반드시 사람 승인 후 전달

## 2. 마스터 하네스 구조

```
[의뢰서 입력]
    │
    ▼
Inquiry Intake Agent          ← 텍스트/CSV/JSON/수동 입력 정규화
    │
    ▼
Inquiry Analysis Agent        ← 핵심 요약, 요구사항 추출, 누락 정보 탐지
    │
    ├──────────────────────────────────────┐
    ▼                                      ▼
Task Classification Agent         Consultation Agent
(작업 유형·기술스택·난이도)         (상담질문·메시지초안·내부메모)
    │                                      │
    ▼                                      ▼
Estimate Support Agent            Risk Review Agent
(견적·기간·범위 보조)               (납기·보안·약관·개인정보 리스크)
    │                                      │
    ▼                                      ▼
Automation Spec Agent             Message Draft Agent
(개발 사양서 생성)                   (채널별 메시지 초안)
    │                                      │
    ▼                                      ▼
Claude Prompt Agent               Approval Agent       ← 사람 승인 단계
(구현 프롬프트 생성)                (승인/반려/수정 관리)
    │                                      │
    ▼                                      ▼
Codex Validation Agent        Messenger Delivery Agent
(검증 프롬프트·결과 반영)           (발송 대기·완료·실패 로그)
    │
    ▼
Delivery Package Agent        ← 고객 제공 패키지
    │
    ▼
LLM Wiki Agent                ← 전체 상태 기록 + handoff 생성
```

## 3. 서브에이전트 업무 분장

### Inquiry Intake Agent
- 입력: 의뢰서 텍스트, CSV, JSON, 수동 입력
- 출력: 정규화된 의뢰 데이터 (표준 스키마)
- 연결: Inquiry Analysis Agent

### Inquiry Analysis Agent
- 입력: 정규화된 의뢰 데이터
- 출력: 요구사항 요약, 핵심 쟁점, 누락 정보
- 연결: Task Classification Agent, Consultation Agent

### Task Classification Agent
- 입력: 요구사항 요약
- 출력: 작업 카테고리, 기술스택 후보, 난이도
- 연결: Automation Spec Agent, Estimate Support Agent

### Consultation Agent
- 입력: 의뢰 분석 결과
- 출력: 상담 질문, 제안 메시지, 내부 메모 (고객용/내부용 분리)
- 연결: Message Draft Agent, Risk Review Agent

### Estimate Support Agent
- 입력: 작업 분류 결과, 요구사항, 리스크
- 출력: 예상 견적 범위, 예상 기간, 범위 경계, 전제 조건
- 연결: Consultation Agent, Proposal Agent

### Risk Review Agent
- 입력: 의뢰 분석 결과, 상담 메시지, 개발 사양
- 출력: 리스크 목록, 차단 항목, 수정 권고
- 연결: Approval Agent, Codex Validation Agent

### Message Draft Agent
- 입력: 상담 전략, 고객 상태, 전달 목적
- 출력: 카카오톡/텔레그램/이메일 채널별 메시지 초안
- 연결: Approval Agent

### Approval Agent
- 입력: 메시지 초안, 리스크 검토 결과
- 출력: 승인 대기 / 승인 완료 / 반려 / 수정 요청
- 연결: Messenger Delivery Agent, LLM Wiki Agent

### Messenger Delivery Agent
- 입력: 승인된 메시지, 채널 정보
- 출력: 발송 대기(복사 가능 형태) / 발송 완료 / 실패 로그
- 연결: Delivery Log Agent

### Automation Spec Agent
- 입력: 작업 분류 결과, 상담 결과, 고객 요구사항
- 출력: 개발 사양서, 기능 목록, 파일 구조 후보
- 연결: Claude Prompt Agent

### Claude Prompt Agent
- 입력: 개발 사양서
- 출력: Claude Code 구현 프롬프트
- 연결: Claude Code 구현 플로우

### Codex Validation Agent
- 입력: 개발 사양서, 구현 프롬프트, 산출물
- 출력: 검증 프롬프트, 검증 결과, 수정 요청
- 연결: Claude Prompt Agent (검증 실패 시 되돌림), LLM Wiki Agent

### Delivery Package Agent
- 입력: 검증 완료 산출물, 사용법, 설치법
- 출력: 고객 제공 패키지, 안내문, 납품 체크리스트
- 연결: Message Draft Agent, Approval Agent

### LLM Wiki Agent
- 입력: 전체 처리 상태, 결정 사항, 검증 결과
- 출력: LLM Wiki 문서 갱신, handoff-prompt.md
- 연결: 모든 에이전트

## 4. 작업 흐름 (Work Flow)

```
1. 의뢰서 입력 (텍스트/CSV/JSON/수동)
2. 입력 정규화
3. 의뢰서 분석 (요약·쟁점·누락 탐지)
4. 개발 작업 분류 (카테고리·기술스택·난이도)
5. 상담 질문 생성 + 고객 응대 메시지 초안 생성
6. 견적/기간/범위 보조 판단
7. 리스크 검토 (납기·보안·약관·개인정보)
8. 채널별 메시지 변환 (카카오톡/텔레그램/이메일)
9. 발송 전 승인 대기 → 사람 검토
10. 승인 시: 복사 가능한 메시지 출력 (MVP) / 자동 발송 준비 (확장)
11. 자동화 개발 사양서 생성
12. Claude Code 구현 프롬프트 생성
13. Codex Goal Mode 검증 프롬프트 생성
14. Codex 검증 실행
    - 실패 시 → 개발 사양 또는 프롬프트 수정 후 재검증
    - 성공 시 → 다음 단계 진행
15. 고객 제공 패키지 생성
16. 고객 전달 메시지 생성
17. 최종 승인 대기 → 납품
18. LLM Wiki 기록
19. handoff-prompt.md 갱신
```

## 7. 메신저 전달 구조

### 카카오톡
- MVP: 승인된 메시지 복사 가능한 텍스트 출력
- 확장: 공식 비즈니스 API 확인 후 자동 발송 검토

### 텔레그램
- 내부 운영 알림 채널 우선 (봇 토큰/채팅 ID → 환경변수)
- 고객 전달은 승인 후 발송 구조

### 이메일
- SMTP 또는 이메일 API (Adapter 구조)
- 제목·본문·첨부·수신자·참조·발송 승인 상태 분리
- 발송 전 미리보기 + 승인 단계 필수

### 공통 원칙
- 고객 메시지 ≠ 내부 알림 (반드시 분리)
- 발송 실패 시 → 실패 로그 기록 후 재검토 흐름
- 오발송 방지: 발송 전 수신자 확인 단계 포함

## 15. 실제 착수용 작업지시서

### Phase 1 — 기반 구조 (MVP 착수)
1. 파일 구조 생성 (`src/`, `llm-wiki/`, `prompts/`, `data/samples/`, `tests/`)
2. 의뢰서 입력 스키마 작성 (`src/types/inquiry.ts` 또는 `.py`)
3. 의뢰서 분석 결과 스키마 작성
4. 개발 작업 분류 스키마 작성
5. 상담 메시지 출력 스키마 작성
